# Audiobook Studio — Agent 项目规范手册

> **适用对象**：所有参与本项目开发的 AI Agent 和开发人员。
> **生效时间**：自本文件创建起永久生效。
> **维护要求**：如规范需修改，必须在本文档中更新并提交 `docs:` commit。

> **相关规范**：
> - `HARNESS_SPECIFICATIONS.md` — LLM 参与各环节的马具规范、架构、逻辑链与自动迭代机制（**必读**）
> - `PROJECT.md` — 项目业务与流水线说明
> - `CONTRIBUTING.md` — 社区贡献流程

## 一、开发准则（必须遵守）

### 1.1 操作边界

| 规则 | 说明 |
|------|------|
| **仅限项目目录** | 仅拥有当前目录 (`/Users/guwj/Desktop/AI_Lab/audiobook`) 及其子目录的读写授权，严禁越界操作 |
| **禁止越权** | 严禁修改、移动或删除本项目文件夹以外的任何文件 |

Agent 可根据目标完全自主运行开发命令，直至项目目标达成。

## 二、提交信息规范

使用 **Conventional Commits** 前缀，每次提交必须使用以下之一：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 添加 EPUB 文本提取` |
| `fix:` | 错误修复 | `fix: 修复中文 PDF 提取乱码` |
| `docs:` | 文档更新 | `docs: 补充 Docker 部署说明` |
| `refactor:` | 代码重构（不改功能） | `refactor: 优化文本分段算法` |
| `test:` | 添加/修改测试 | `test: 添加 TTS 合成单元测试` |
| `chore:` | 其他杂务 | `chore: 更新 Python 依赖版本` |

> 不确定时使用 `chore:` 前缀。

## 三、代码质量规范

### 3.1 代码风格工具

所有代码风格检查通过 `.pre-commit-config.yaml` 自动执行：

| 工具 | 用途 |
|------|------|
| **Black** | 自动格式化 |
| **isort** | 导入排序 |
| **flake8** | PEP8 检查 |
| **bandit** | 安全审计 |
| **detect-secrets** | 密钥泄露检测 |

### 3.2 质量保障

- 单元测试覆盖率 **≥ 80%**（`pytest --cov=src`）
- 所有 Python 文件提交前必须通过语法检查
- 禁止提交包含 `print("debug")` 等调试语句的代码
- 禁止提交大块注释掉的代码（除非有明确说明）
- 文件中禁止尾随空格和制表符混用

---

## 四、环境与安全规范

### 4.1 密钥管理

| 规则 | 说明 |
|------|------|
| **绝对禁止** | 在代码中硬编码 API 密钥、密码或 Token |
| **禁止提交** | `.env`、`config_real.py`、`keys.json`、`*.key`、`*.secret` |
| **正确做法** | 复制 `.env.example` 为 `.env`，填入真实值 |
| **泄露应急** | 立即运行 `git log -p --grep="API\|key\|secret\|token"` 检查历史，更换对应密钥 |

### 4.2 环境变量模板

所有新环境变量必须在 `.env.example` 中添加注释说明。

## 五、状态管理与容错

### 5.1 断点续传

- 长时间任务自动在 `checkpoints/` 保存进度快照
- 网络中断后重新运行相同命令，自动从断点继续
- Agent 实现代码时必须支持断点恢复逻辑

### 5.2 LLM 服务容错

- 多 LLM 提供商自动轮换（Gemini、Groq、NVIDIA、OpenRouter）
- 所有服务不可用时保存进度并暂停，而非崩溃
- Agent 实现 LLM 调用时必须包含重试逻辑和降级策略

## 六、日志规范

- 所有日志保存到 `logs/` 目录，按日期命名
  - `logs/YYYY-MM-DD_main.log` — 主程序日志
  - `logs/YYYY-MM-DD_errors.log` — 仅错误和警告
- 日志必须包含时间戳和清晰的错误层级
- Agent 实现代码时必须使用项目统一的 `logger` 模块，禁止自行 `print`

## 十、应急处理

| 情况 | 处理方式 |
|------|---------|
| 提交被 pre-commit 阻止 | 按照错误提示逐项修复后重新提交 |
| 不确定提交前缀 | 使用 `chore:` 安全选项 |
| 担心密钥泄露 | 立即检查 git 历史，更换密钥 |
| CI 流水线失败 | 检查 GitHub Actions 日志，修复后重新推送 |
| LLM 服务持续不可用 | 检查网络和 API 额度，考虑添加备用密钥 |

## 十一、开发模式例外条款（Agent 专用）

> **目的**：在 Agent 开发/原型阶段，部分规范可临时放宽以提升迭代速度。
> **生效条件**：当前工作在 `feature/*` 或 `bugfix/*` 分支，且 PR 尚未合并到 `develop`。
> **提交要求**：合并到 `develop`/`main` 前必须回归全部规范。

### 11.1 可以放宽的规则

| 规范原文 | 开发模式放宽为 | 合并前必须恢复 |
|---------|--------------|--------------|
| **§3.2** 禁止 `print("debug")` | 允许临时 `print` 调试，但须在 final commit 前清理 | ✅ 用 `grep -rn "print(" src/` 检查清理 |
| **§3.2** 禁止提交大块注释代码 | 允许保留功能相关的注释代码（附 `TODO:` 标记），允许临时注释 | ✅ `git diff develop...HEAD \| grep "^\+.*#\s*TODO"` 清零 |
| **§3.2** 禁止尾随空格 | 允许，pre-commit 中的 `trailing-whitespace` 钩子可自动修复 | ✅ pre-commit 自动处理 |
| **§五** 必须实现断点续传 | 允许先实现核心逻辑，断点续传作为 TODO 延后 | ✅ 须有对应 issue 编号 |
| **§五** LLM 调用必须重试+降级 | 允许 MVP 阶段先用单厂商+单次调用，重试逻辑延后 | ✅ 须有对应 issue 编号 |
| **§六** 强制使用 logger 禁止 print | 允许开发阶段用 `print`，final commit 前替换为 `logger` | ✅ `grep -rn "print(" src/` 清零 |
| **§七** 每次改代码必须更新文档 | 允许在功能完成时一次性批量更新文档 | ✅ 分支合并前文档已更新 |
| **§九** 每步立即记录 PROJECT.md | 允许完成一个完整功能点后批量记录 | ✅ 分支合并前已记录 |

### 11.2 仍须严格遵守的规则（不可放宽）

| 规则 | 原因 |
|------|------|
| **§1.1** 仅限项目目录操作 | 安全红线，无例外 |
| **§1.2** 核心保护区严禁干预 | 系统稳定，无例外 |
| **§4.1** 禁止硬编码密钥 | 安全红线，无例外 |
| **§二** Conventional Commits | 提交历史整洁，无例外 |
| flake8（通过 pre-commit 自动修复） | 代码质量基线 |
| Black + isort 自动格式化 | 风格统一，无额外成本 |

### 11.3 如何启用开发模式

在 feature 分支开发时，Agent 自动适用上述放宽规则。合并前需要通过：

```bash
# 最终清理检查
grep -rn "print(" src/ --include="*.py" || echo "已清理"
grep -rn "#\s*TODO" src/ --include="*.py" || echo "无遗留 TODO"
```

> **核心原则**：系统会自动检查和执行规范，Agent 只需遵循本文档要求开发即可。遇到本文档未覆盖的情况，优先选择最安全、最保守的方案。
