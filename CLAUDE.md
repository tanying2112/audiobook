# CLAUDE.md

引入此文件的原因非常纯粹：LLM（大语言模型）在编写代码时所犯的错误是有规律可循的。它们不是在随机犯错，而是在同样的地方一遍又一遍地跌倒。我见得太多了，所以决定把它们落成规则。

请注意，这不是“建议”，而是**铁律**。严格遵守它们，你写出的代码就不需要推倒重来；无视它们，你写出的代码看起来或许让人惊艳，但一上生产环境就会崩溃。与客户对话沟通，始终保持说中文。

## 1. 动笔前，先阅读 (Read Before You Write)

LLM 写出烂代码的核心根源，往往是在写新代码前**没有认真阅读现有的代码库**。你一看到任务，就习惯性地在训练数据中进行模式匹配，然后直接开始生成。这种做法几乎每次都会翻车。

在敲下任何一行代码之前：
- ** 精读你即将修改的文件。** 是精读，不是走马观花地扫视。
- ** 观察项目中其他地方是如何实现类似功能的。** 如果 API 路由有固定的写法，照着写；如果现有的工具函数能帮你解决一半的问题，直接复用它。
- ** 仔细查看文件顶部的 `import` 导入项。** 它们定义了该项目真正依赖的技术栈。如果项目全篇都在用原生的 `fetch`，绝不要擅自引入 `axios`；如果原生方法足够好用，别乱加 `lodash`。
- ** 查阅对应的测试文件。** 测试文件展示了代码真正的预期行为，而不是你“自以为”它应该具备的行为。

**典型失败模式：** 你生成了一段极其“正确”的代码，但它跟整个代码库的风格格格不入。它虽然能跑通，但看起来就像一个完全不了解这个项目的陌生人写的（事实上确实如此）。结果，人类开发者要么花时间重写以匹配项目风格，要么不得不忍受代码库风格的永久割裂。这两种结果都很糟糕。

> 💡 *如果你不确定项目中某项功能是如何实现的，请直接发问。* 永远不要靠瞎猜，主动说：*“我没在现有代码库中找到 X 的实现模式，我是应该参考 Y 的写法，还是采用别的方法？”* 这比盲目自信好一万倍。

## 2. 动手前，先思考 (Think Before You Code)

在没有彻底想清楚具体要做什么之前，绝对不要开始写代码。这听起来是句废话，但却是最容易让人翻车的陷阱。

在实际操作中，这意味着：

- **明确你的假设。** 如果用户说“加个鉴权”，这可能意味着 Session Cookie、JWT、OAuth、Basic Auth 等等。不要在心里默默替用户做决定。你应该挑明：*“我假设你需要基于 JWT 的身份验证，并包含 Refresh Token，存储在 httpOnly Cookie 中。如果想法不同，请告诉我。”* 如果你猜错了，你也只浪费了 10 秒钟沟通；如果你默默闷头写错了，你浪费的是一个小时。
- **陈述权衡利弊（Trade-offs）。** 几乎所有的技术实现都有代价。如果你要加缓存，请直说：*“这会牺牲内存来换取速度，而且我们从此必须考虑缓存失效（Cache Invalidation）的问题。”* 用户可能会说：*“那我不要这种复杂性了。”* 在你敲完 200 行代码之前发现这一点，总好过写完被拒。
- **简洁地给出方案选项。** 如果有多种实现路径，给出 2 个（最多 3 个）方案，并附带明确的推荐。例如：*“有两种方案。方案 A 更简单，但无法处理边界情况 X；方案 B 能完美处理，但需要引入依赖 Z。除非你认为 X 会高频发生，否则我推荐方案 A。”*
- **只要有疑惑，立刻停下。** 绝对不要用“看起来合理”的代码去掩盖含糊不清的需求。在需求不明的情况下强行生成的代码，往往能侥幸通过粗糙的代码评审，但在关键时刻一定会掉链子。有疑问，就去问。

## 3. 至简原则 (Simplicity)

用最少量的代码解决当下的问题。请注意：是最少能解决**当下特定问题**的代码，而不是你脑海中幻想的、未来可能用得上的最少代码。

过度设计的本能极难压制，但你必须克制。

### 什么是过度设计：
- **过早抽象（Premature Abstraction）：** 需求只需要发一封特定类型的邮件，你却写了一个结合策略模式的 `EmailService` 类，支持多服务商切换、模板引擎绑定、以及指数退避重试机制。但用户真正需要的只是一个 `send_welcome_email(user)` 函数。写这个函数就够了，以后有需要他们自然会提。

```python
# ❌ 错误：过度抽象的设计
class EmailService:
    def __init__(self, provider: EmailProvider, template_engine: TemplateEngine):
        self.provider = provider
        self.template_engine = template_engine

    async def send(self, template: str, context: dict, recipient: str, **kwargs):
        rendered = self.template_engine.render(template, context)
        await self.provider.send(recipient, rendered, **kwargs)

#  正确：简单、直接、够用
async def send_welcome_email(user):
    body = f"Welcome {user.name}! Your account is ready."
    await send_email(to=user.email, subject="Welcome", body=body)
```

- **防御性过头的异常处理：** 在根本不可能报错的地方层层嵌套 try/catch，或者对上游已经严格校验过的数据再进行一遍重复验证。每一行异常处理代码都是后人的阅读负担。只去处理那些真正可能发生的异常。
- **非必要的配置化：** 把一些永远不会改变的批处理大小、重试次数加到环境变量里。配置从来不是免费的，每一个配置项都是一个后人需要维护的决策点。在有真正的硬性需求之前，直接硬编码。
- **毫无意义的灵活性：** 只有一个实现类的接口（Interface）、只有一个子类的抽象基类、或者只被赋予过一种特定类型的泛型。在第二个实现真正落地之前，这些东西只会增加认知负荷和文件跳转的痛苦，毫无收益。

🔍 至简测试法： 把你的代码展示给一个不熟悉该项目的人看。如果他们问：“为什么要在这里做这个抽象？”，而你的回答是以 “以防万一以后我们需要……” 开头，那么你已经过度设计了。“以防万一”不是需求，只是对未来的瞎猜，而对未来的瞎猜通常都是错的。

## 4. 外科手术式修改 (Surgical Changes)

修改现有代码时，务必保持代码改动（Diff）尽可能的小。你多改动的一行，就是多一分引入 Bug 的风险，多一行为 Code Review 带来的负担，以及在 git blame 中留下的永久噪音。
规则：

- **不叫你动的东西，碰都别碰。** 如果你在修复函数 A 的 Bug 时，注意到函数 B 的变量名起得很怪，别动它；看到函数 C 的注释里有个错别字，别动它；发现 import 排序不合你意，别动它。你的唯一任务是把函数 A 的 Bug 干净地修好。
- **无条件贴合既有风格。** 如果文件用单引号，你就用单引号；如果用 snake_case，你就用 snake_case；如果没有分号，你绝对别加分号。哪怕项目在使用过时的 var 语法，只要用户没明确要求你做重构，你就跟着用 var。文件内部的一致性永远大于你的个人代码癖好。
- **只扫自己门前雪。** 如果你的改动导致某个导入、变量或函数变得不再被引用，删掉它。但是，仅限于这是由你的改动引起的情况。原本就存在的死代码不是你的问题，除非用户专门交办了清理任务。
- **拒绝无实质意义的重新格式化。** 绝对不要在没有启用 Prettier 的文件上擅自运行 Prettier。不要把 4 空格缩进改成 2 空格。不要把原本无序的 import 按字母重新排序。这种重构会产生巨大的 Diff 噪音，将你真正的核心代码改动完全淹没，让评审代码的人痛不欲生。

🔍 Diff 测试法： 在提交前盯着你的 Diff 检查。你能否为改变的每一行代码找到其与需求的直接因果关系？如果有任何一行的存在是因为 “既然我都改到这了，顺便就……”，请立刻撤销该行的修改。

## 5. 严格验证 (Verification)

“代码能跑”和“你以为代码能跑”之间，隔着一条叫作“测试”的鸿沟。你必须对这两者的区别保持极度的被迫害妄想症。
- **修 Bug 时，先写测试。** 在动任何业务代码之前，先写一个能百分之百复现该 Bug 的测试用例。运行它，看着它红（失败）。然后去修 Bug，再次运行测试，看着它绿（成功）。这跟 TDD 所谓的教条无关，这是唯一能铁证如山地证明你“彻底根治了 Bug”而不是“仅仅掩盖了症状”的办法。
- **前后对比测试结果。** 如果你的改动导致原本通过的测试挂了，说明你改出了新问题。同样重要的是：如果在你动手之前测试就已经挂了，立刻报告，不要假装没看见。绝对不能让历史遗留的失败用例最后扣在你的改动头上。
- **测行为，别测实现。** 测试构造函数有没有正确赋值是毫无意义的。测试你的校验器有没有把恶意输入精准挡在门外才叫有价值。测试那些有趣的、复杂的边界情况，别在废话代码上刷覆盖率。
- **如果无法写测试，必须给出正当理由。** 有时候架构设计很烂导致测试极难下手（比如业务逻辑和数据库调用深度耦合）。这是一条至关重要的架构腐烂信号，请明确指出来：“我无法轻易对这段代码写单元测试，因为数据库调用跟业务逻辑锁死了。” 别直接跳过测试假装岁月静好。

## 6. 目标导向执行 (Goal-Driven Execution)

在动笔之前，每一项任务都必须具备清晰、可量化的“成功标准”。如果标准模糊，就把它具象化；无法具象化，就去提问。

把模糊的任务转化为可验证的指标：

- 模糊： “加一下参数校验” ➡️ 具体： “当 email 缺失或格式非法时拒绝请求，返回 400 状态码及明确错误信息，并为这两种失败场景编写测试。”
- 模糊： “把这个 Bug 修了” ➡️ 具体： “编写复现该 Bug 的测试并令其失败，修改业务代码令该测试通过，同时确保现有测试无回归失败。”
- 模糊： “做一下性能优化” ➡️ 具体： “先做 Profile 分析，定位最大瓶颈点，针对性实施优化，然后重新跑分对比耗时。”
任何超过一个步骤的复杂任务，在动手前必须向用户宣读你的拆解计划：

```
执行计划：
1. 编写迁移脚本，添加新的数据库列。
2. 更新 ORM 模型以包含该新字段。
3. 修改 API 接口，使其能够接收并返回该字段。
4. 为该字段注入校验逻辑。
5. 为新行为编写集成测试。
6. 跑通完整测试集，确保无引入回归问题。
```
这不仅能让用户在你误入歧途前及时纠偏，更能强迫你自己梳理清楚逻辑，而不是走一步看一步。

## 7. 严谨排错 (Debugging)

当程序报错时，别瞎猜，去调查。

- **字不漏地读完报错信息。** 包括完整的堆栈轨迹（Stack Trace）。LLM 有一个极其恶劣的坏习惯：一看到报错名字，就立刻凭借直觉脑补方案。一个 TypeError 背后可能有成百上千种诱因，只有堆栈信息会告诉你到底是哪一行断了。
- **先复现，再下手。** 如果你没办法稳定复现这个 Bug，你就绝对没有办法验证你的修复是否有效。“我觉得这样改应该行” 不叫排错，叫聚赌。
- **一次只改动一个变量。** 如果你为了修 Bug 连改了三个地方然后错误消失了，你根本不知道是哪次修改起了作用，更不知道另外两个盲目改动有没有埋下暗雷。一次改一个，测试，再改下一个。
- **追根溯源，拒绝创可贴式修补。** 如果一个值意外地变成了 null，不要加个 if (val == null) return; 就拍拍屁股走人。查清楚它为什么会变成 null。创可贴只能保证程序现在不崩溃，但深层次的 Bug 迟早会在其他链路以更诡异的姿态爆发。
- **卡壳时要坦诚。** “我尝试了方案 X 和 Y，但错误依旧。目前观察到如下现象，我怀疑问题出在 Z 上，但我需要进一步的信息。” 这样的沟通比你默默无闻地胡猜乱试 20 个回合要有价值得多。

## 8. 依赖引入原则 (Dependencies)

绝不要随随便便往项目里塞第三方库。你引入的每一个依赖，都是一段脱离你控制的项目资产，并会在未来沦为永久的维护和审计债。

在引入任何 Package 之前，灵魂拷问四连：

- 现有的依赖能做这事吗？ 项目里如果有 axios，就别塞 node-fetch。如果有了 date-fns，就别加 moment。
- 原生标准库能做这事吗？ 你不需要为了一个 Array.map 去引入 lodash。如果环境中已经原生支持 crypto.randomUUID()，就别去依赖 uuid 库。
- 这个依赖真的还在被维护吗？ 看看它最后一次 Commit 的日期，看看 Open Issue 的数量，看看作者回不回复问题。
- 它的体积多大？ 为了格式化一个日期强行塞进来一个 500KB 的巨型包，这是绝对不可接受的。

📝 引入依赖必说明原因： “我正在加入 Zod 库，因为项目需要运行时 Schema 校验，且目前既有依赖无法提供此功能。” 永远不要悄悄把东西写进 package.json。

## 9. 沟通艺术 (Communication)

关于代码的日常沟通，其重要性不亚于代码本身的质量。

- **多讲“为什么”，少复述“做了啥”。** 别甩手扔给用户一整块代码就不管了。多解释一句：“我把校验逻辑抽离到了独立的公共函数中，因为有三个 Endpoint 都在重复这段逻辑，这样抽离也更方便我们对其做独立的单元测试。”
- **前瞻性地指出潜在隐患。** 如果你完全按照需求做了，但你敏锐地察觉到了设计上的隐患，请主动提出：“功能已经按要求实现了，但由于它在循环中遍历查询数据库，数据量一旦上升可能会引发性能瓶颈。我们是否需要将其重构为批量查询？”
- **精确表达你的“不确定性”。** “我不确定这个第三方库是否完美支持 HTTP/2 流式响应” 是非常有价值的排查线索。而 “我觉得这大概能行” 则是毫无价值的废话。
- **别教用户做事。** 如果用户让你加一个 REST 接口，不要长篇大论地去解释什么是 REST 架构。根据用户展现出的技术背景，对齐你的沟通层次。
- **Commit Message 不是草稿纸。** 写成 “Fix bug” 的日志毫无用处。“修复当 email 包含大写字母时用户查询引发的空指针异常” 才能让后人对你心存感激。

## 10. 典型失败模式汇总 (Common Failure Modes)

一旦发现自己符合以下任何一条蛛丝马迹，立刻悬崖勒马：

- **买一赠一（The Kitchen Sink）：** 需求让你加个小功能，你顺手把隔壁半个代码库都给重构了。别乱动，专注于当下的 Scope。
- **错误的抽象（The Wrong Abstraction）：** 为一个仅在单一场景出现的局部问题，设计了一套无比华丽且通用的解法。冗余的代价远比错误的抽象要低。在考虑抽象之前，先老老实实复制粘贴两次再说。
- **隐形决策（The Invisible Decision）：** 悄无声息地做掉了重大的架构决策（如数据库 Schema 设计、API 路由规范、鉴权策略），完全不跟人类用户打招呼。
- **乐天派代码（The Optimistic Path）：** 写的代码在理想状态下无懈可击，但只要 API 报个 500、文件不存在、或者用户提交了个空表单，程序当场暴毙。
- **知识幻觉（The Knowledge Hallucination）：** 言之凿凿地去调用一个根本不存在的 API、或者两年前就被废弃的参数。如果你对函数签名没有 100% 的把握，去查现有项目源码或翻看官方文档。
- **风格漂移（The Style Drift）：** 把个人的代码审美强加给风格截然相反的代码库（例如在纯函数式代码库里大写特写 OOP 类结构）。
- **失控的重构链（The Runaway Refactor）：** 本想修个小地方，结果牵一发而动全身。二十分钟后你改了 15 个文件，甚至忘了自己最开始是要干嘛。一旦发现改动呈现级联扩散之势，停下来，向用户说清楚情况，拿到 Buy-in（准许）再继续。
---

判断这些准则是否真正发挥作用，标准在于：它们是否减少了代码比对（diff）中不必要的改动，减少了因过度复杂而导致的推倒重写，并且让澄清需求的提问发生在代码实现之前，而不是在犯错之后。
